# ReAct variant interpretation agent with deterministic fallback

A tool-using ReAct agent that interprets called variants against local ClinVar/gnomAD data, applies ACMG/AMP 2015 combining rules, and produces guardrailed classification reports. The implementation spans ~6,000 lines across a new `ai-report/agent/` package: VCF parser, 5-tool registry, multi-provider LLM abstraction (Ollama/OpenAI/Anthropic/deterministic), a pure-function ACMG rule engine, report generator with code-enforced safety constraints, and an observability trace layer. A `DeterministicBackend` pre-scripts the tool-call sequence so the full pipeline runs in CI with no network, no GPU, and no non-determinism.

Watch for: (1) The fallback condition in `interpret.py` only re-classifies when the agent's result is still VUS *and* fallback was triggered — a variant that the agent misclassified as Benign after a loop won't be corrected (confirmed). (2) The `_TREATMENT_PATTERNS` regex matches "diagnosed" in legitimate ClinVar condition descriptions, creating false-positive scrubs in summaries (likely). (3) Multi-tool-call responses from real LLMs are handled inside a `for tool_call in response.tool_calls` loop, but messages are appended per-tool inside that loop — if a real model returns 2+ tool calls, conversation history becomes malformed (confirmed). (4) The `KnowledgeBase` connection is never closed in `ToolRegistry` or `ReActAgent` (confirmed).

## High-level view

The ACMG combining rules are duplicated — once as a standalone function in `deterministic.py` (`classify_by_acmg_rules`) and again as a tool implementation in `tools.py` (`_classify_acmg`). They use the same logic but are maintained separately. Drift between these two copies would produce different classifications depending on the code path.

The ReAct loop's multi-tool-call handling has a structural issue: it iterates over all tool calls in a response, appending messages inside the inner loop, but never re-queries the LLM between them. For the deterministic backend (which always returns exactly one tool call) this is invisible, but a real model returning parallel tool calls would produce a malformed conversation history with duplicate assistant turns.

Safety enforcement is layered: regex scrubbing in `enforce_safety_constraints`, evidence-required validation in `final_answer`, VUS flagging in the report renderer, and a mandatory banner inserted programmatically. The guardrail validation function (`enforce_report_guardrails`) is callable in CI as a regression gate. However, the regex scope is too broad — it will scrub legitimate database-sourced text containing clinical terminology.

The fallback gating in `interpret.py` is the highest-risk correctness issue. When a real LLM produces a classification and *then* triggers the loop detector on the next iteration, the fallback fires but the condition `classification == "Uncertain Significance"` prevents re-classification. The agent's potentially-hallucinated answer ships as-is.

1. **Fallback condition too narrow** — `interpret.py` only re-classifies via deterministic when `fallback_triggered AND classification == "Uncertain Significance"`. If the agent loops and produces a wrong non-VUS classification before triggering fallback, it won't be corrected. Widen the condition to trigger on `fallback_triggered` alone, or at least on `fallback_triggered AND confidence == "low"`. (confirmed)
2. **ACMG rule duplication** — `classify_by_acmg_rules()` in `deterministic.py` and `_classify_acmg()` in `tools.py` implement the same combining rules independently. Extract to a single shared function to prevent drift. (confirmed)
3. **Multi-tool-call conversation corruption** — The inner `for tool_call in response.tool_calls` loop appends messages per-call without re-sending to the LLM. With a real backend returning 2+ parallel tool calls, conversation state diverges. Restructure to either process one tool call per iteration or batch-append all tool results before the next LLM turn. (confirmed)
4. **KnowledgeBase connection leak** — `ToolRegistry` and `ReActAgent` never call `kb.close()`. Add a `close()` method or use a context manager. (confirmed)
5. **Regex false-positive on "diagnosed"** — `\bdiagnos(?:e|is|ed)\b` will match "diagnosed" in ClinVar condition descriptions inserted verbatim into summaries. Narrow the pattern or allowlist database-sourced text. (likely)
6. **Untested fallback edge case** — The `fallback_triggered AND not VUS` scenario has no test coverage. Add a test where the agent produces a non-VUS classification then triggers fallback. (confirmed)

## Fallback gating logic in interpret.py

The fallback re-classification in `interpret.py` checks:

```python
if result.fallback_triggered and result.classification == "Uncertain Significance":
    fallback_result = fallback.run(result.variant)
```

If the agent enters a loop, produces a garbage "Benign" classification from a hallucinated `final_answer` call, and *then* triggers the loop-detection guard on the next iteration, the result already has `classification = "Benign"` with `fallback_triggered = True`. The deterministic path would never run. The safe fix is to trigger fallback on `fallback_triggered` unconditionally — if the agent couldn't complete cleanly, the result is suspect regardless of what it managed to emit before failing.

## ACMG rule engine duplication

`deterministic.py::classify_by_acmg_rules` and `tools.py::_classify_acmg` both implement the full ACMG combining rule matrix. They return slightly different shapes (tuple vs. dict) but share identical classification logic. `_classify_acmg` also loads the ACMG criteria JSON that is never actually consulted in the combining logic, suggesting the intent was to validate codes against known criteria — but that validation doesn't happen. The core rule engine should be factored out into a single function both callsites import.

## Multi-tool-call conversation model

In `react.py`, the agent loop structure is:

```
for iteration in range(max_iterations):
    response = backend.generate(messages, ...)
    for tool_call in response.tool_calls:  # inner loop
        tool_result = registry.invoke(...)
        if tool_call == "final_answer" and success:
            return result
        messages.append(assistant_message)   # duplicated per tool call
        messages.append(tool_result_message)
```

When `response.tool_calls` has length > 1, the second append of an assistant message creates a duplicate assistant turn. OpenAI's API expects a single assistant message with all tool calls, followed by one tool-result message per call. The current structure produces `[assistant+call1, tool1, assistant+call1, tool2]` instead of `[assistant+[call1, call2], tool1, tool2]`. With the deterministic backend this never triggers (always 1 tool call), but real LLMs can return multiple.

## Safety regex scope

The pattern `\bdiagnos(?:e|is|ed)\b` fires on any use of "diagnosed", including passive descriptions from ClinVar like "Condition: Prion disease, first diagnosed in childhood." Because ClinVar record text is woven into the generated summary by the deterministic template (`_generate_summary`), and ClinVar conditions are inserted verbatim via `clinvar_conditions`, this regex could scrub legitimate provenance text. The risk is low today because the chr20 knowledge base has short condition strings, but it would surface with a larger knowledge base.

## Test coverage gaps

Not tested: multi-tool-call handling (invisible with deterministic backend), the fallback gating logic in `interpret.py` for the `fallback_triggered AND not VUS` case, and `_classify_acmg`'s unused ACMG criteria JSON load (the JSON is fetched but never validated against).
